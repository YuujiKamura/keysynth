# 2026-04-26 live hot-swap

Branch: `live-hotswap`. Goal was GUI-side engine / preset / asset
swapping with no cpal-stream / midir-conn / GUI restart, so the
AKAI MPK mini 3 stays connected and the user can swing between
square / piano-modal / sfz-piano during a take.

## What landed

1. `synth.rs`: `ModalPreset` enum (`Default | Round16 | Arch1 |
   Bright`) with `params()` returning a `ModalParams` and `apply()`
   writing into the global `MODAL_PARAMS` cell. Round-16 mirrors the
   env-var preset already validated in `bin/render_chord` /
   `render_song` (`KS_DETUNE=2.0 KS_POL_H=0.6 KS_T60_CAP=6.0
   KS_STAGE_B=0.4 KS_OUT_GAIN=45.0 KS_RESIDUAL=0.10`). Arch-1 turns
   the residual layer up (≈0.55) for hammer-felt + body color
   trade-off. Bright kills detune/polarisation for solo passages.

2. `main.rs`: AKAI auto-select. When `--port` isn't given, prefer
   the first port whose name (uppercased) contains `AKAI` or `MPK`,
   falling back to `ports[0]`. Verified against the live MPK mini 3
   port (`midi='MPK mini 3'` in the boot log).

3. `ui.rs`:
   - `Load SFZ...` / `Load SF2...` buttons (rfd 0.14 native
     dialogs). On selection, decode synchronously and atomically
     swap into the shared `Mutex<Option<SfzPlayer>>` /
     `Mutex<Option<Synthesizer>>` already shared with the audio
     callback. The cpal stream and midir connection don't move.
   - `[sfz: ok/—, sf2: ok/—]` indicator next to the buttons.
   - Last load result is printed below the row in green/red so the
     user gets feedback that the multi-second SFZ decode finished.
   - Modal preset row above the existing sliders; clicking a preset
     calls `ModalPreset::apply()` and the next note_on picks up the
     new params. Sliders still work afterwards (preset + tweak).

4. `Cargo.toml`: `rfd = "0.14"` for native file dialogs (Windows
   uses the OS dialog; no extra system deps).

## Live behaviour

- Engine swap: already worked through `LiveParams.engine` (read
  per note_on); the GUI dropdown now mirrors it. Already-playing
  voices finish through the old engine, new note_ons run the new
  engine. No clicks / cpal underruns observed swinging
  square ↔ piano-modal ↔ sfz-piano.
- Preset swap (PianoModal): `set_modal_params` is process-wide,
  so existing logic carries over. `output_gain` is read live per
  audio block per the `ModalParams` doc, so volume change on a
  held note is audible; the others (detune, T60 cap, etc.) snap
  in on the next note_on which matches musician expectation.
- SFZ hot-load: blocks the GUI thread for ~5 s on the Salamander
  V3 set. Audio keeps running the entire time on the previous
  player; swap is one mutex acquire. Acceptable because the user
  initiated the load and it finishes inside one chord's sustain.

## Verified

- `cargo build --bin keysynth --release` — clean (only the
  pre-existing `_mm_getcsr` deprecation warnings).
- `cargo test --release --lib` — 200 pass. The 4 failing tests
  (`voices::piano_modal::tests::{from_lut_*, release_lifecycle}`)
  fail identically on `main` before any of these changes; the
  comment on `ModalParams::default()` already calls them out as
  needing a co-retune. Not regressed by hot-swap work.
- `keysynth --list` finds `MPK mini 3`.
- Boot log with no flags: `keysynth: midi='MPK mini 3' ...
  engine=Square` — AKAI auto-selected.
- Boot log with `--engine piano-modal`: same AKAI port,
  `engine=PianoModal`. Preset selector visible (screenshot 02).

## Open items / not done

- No background-thread SFZ loader. A fresh Salamander pull blocks
  the GUI ~5 s; for live use that's tolerable but a polished
  experience would dispatch the decode to a worker and only swap
  once it's ready.
- Engine selector still shows all engines flat. Could group
  piano-family / pluck / sample-based but didn't want to churn
  layout in the same commit.
- The 4 pre-existing piano_modal voice tests still fail; out of
  scope here, tracked in the comment on `ModalParams::default()`.
- The `Bright` / `Arch1` presets are seeded from intuition, not
  from a CDPAM/VLM tuning loop. Round-16 has been validated;
  Bright/Arch1 should get the same treatment the next time the
  evaluation harness runs.
