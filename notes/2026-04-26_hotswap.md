# 2026-04-26 live hot-swap

Branch: `live-hotswap`. Goal was GUI-side engine / preset / asset
swapping with no cpal-stream / midir-conn / GUI restart, so the
AKAI MPK mini 3 stays connected and the user can swing between
square / piano-modal / sfz-piano during a take.

## What landed

1. `synth.rs`: `ModalPreset` enum (`Default | Round16 | Arch1 |
   Bright`) with `params()` returning a `ModalParams` and `apply()`
   writing into the global `MODAL_PARAMS` cell. Round-16 mirrors the
   env-var preset already validated in `bin/render_chord` /
   `render_song` (`KS_DETUNE=2.0 KS_POL_H=0.6 KS_T60_CAP=6.0
   KS_STAGE_B=0.4 KS_OUT_GAIN=45.0 KS_RESIDUAL=0.10`). Arch-1 turns
   the residual layer up (≈0.55) for hammer-felt + body color
   trade-off. Bright kills detune/polarisation for solo passages.

2. `main.rs`: AKAI auto-select. When `--port` isn't given, prefer
   the first port whose name (uppercased) contains `AKAI` or `MPK`,
   falling back to `ports[0]`. Verified against the live MPK mini 3
   port (`midi='MPK mini 3'` in the boot log).

3. `ui.rs`:
   - `Load SFZ...` / `Load SF2...` buttons (rfd 0.14 native
     dialogs). On selection, decode synchronously and atomically
     swap into the shared `Mutex<Option<SfzPlayer>>` /
     `Mutex<Option<Synthesizer>>` already shared with the audio
     callback. The cpal stream and midir connection don't move.
   - `[sfz: ok/—, sf2: ok/—]` indicator next to the buttons.
   - Last load result is printed below the row in green/red so the
     user gets feedback that the multi-second SFZ decode finished.
   - Modal preset row above the existing sliders; clicking a preset
     calls `ModalPreset::apply()` and the next note_on picks up the
     new params. Sliders still work afterwards (preset + tweak).

4. `Cargo.toml`: `rfd = "0.14"` for native file dialogs (Windows
   uses the OS dialog; no extra system deps).

## Live behaviour

- Engine swap: already worked through `LiveParams.engine` (read
  per note_on); the GUI dropdown now mirrors it. Already-playing
  voices finish through the old engine, new note_ons run the new
  engine. No clicks / cpal underruns observed swinging
  square ↔ piano-modal ↔ sfz-piano.
- Preset swap (PianoModal): `set_modal_params` is process-wide,
  so existing logic carries over. `output_gain` is read live per
  audio block per the `ModalParams` doc, so volume change on a
  held note is audible; the others (detune, T60 cap, etc.) snap
  in on the next note_on which matches musician expectation.
- SFZ hot-load: blocks the GUI thread for ~5 s on the Salamander
  V3 set. Audio keeps running the entire time on the previous
  player; swap is one mutex acquire. Acceptable because the user
  initiated the load and it finishes inside one chord's sustain.

## Open items / not done

- No background-thread SFZ loader. A fresh Salamander pull blocks
  the GUI ~5 s; for live use that's tolerable but a polished
  experience would dispatch the decode to a worker and only swap
  once it's ready.
- Engine selector still shows all engines flat. Could group
  piano-family / pluck / sample-based but didn't want to churn
  layout in the same commit.
- The 4 pre-existing piano_modal voice tests still fail; out of
  scope here, tracked in the comment on `ModalParams::default()`.
- The `Bright` / `Arch1` presets are seeded from intuition, not
  from a CDPAM/VLM tuning loop. Round-16 has been validated;
  Bright/Arch1 should get the same treatment the next time the
  evaluation harness runs.

## 2026-04-26 follow-up: voice browser side panel + Physics preset

User reported `Modal (Arch1)` was clipping under live MIDI — the
`residual_amp = 0.55` in that preset detonated the commuted-residual
path which is now always-on at master HEAD (was env-gated when the
preset was authored). Removed `Arch1` from `ModalPreset` entirely.

Cherry-picked `2033ba9` (`piano_modal: pure-physics constructor`)
from `arch-from-physics` onto `live-hotswap` so the analytical-mode
synthesis path (Stulov hammer + Fletcher inharmonicity + Valimaki
T60) is available in the live binary. Added a process-wide
`MODAL_PHYSICS` atomic flag plus `set_modal_physics()` so the GUI
can flip into the physics path without an env var or restart;
`make_voice` consults `modal_physics_enabled()` (which OR's the
runtime flag with the `KS_PHYSICS` env var so offline renders still
behave the same). Added `ModalPreset::Physics` whose `apply()` both
sets clean `ModalParams` (zero detune / pol / residual — the
analytical generator already produces a complete bank) and flips
the physics flag on. Selecting any other modal preset flips it back
off.

Replaced the flat engine row + asset row + GM 128 left panel with a
**voice browser side panel** (`src/voice_lib.rs` + `SidePanel::left`
in `src/ui.rs`). Three categories — **Piano** (modal default /
round-16 / physics / bright, KS Piano variants, auto-discovered
Salamander SFZ, GeneralUser SF2), **Other (synth)** (square / KS /
sub / FM / koto), **Custom** (user-saved snapshots). Each entry is
a `VoiceSlot { label, category, engine, params, asset_path,
sf_program/bank, modal_physics, builtin }`. Selecting a slot
performs engine swap + ModalParams write + asset hot-load in one
operation; an already-decoded SFZ / SF2 path is detected and the
re-load is skipped (the auto-discovered Salamander loaded at
startup is one of these — first click on `Salamander SFZ` is 0 s).

Per-category action buttons:
- Piano: `+ Load SFZ...`, `+ Load SF2...` (rfd dialog → entry
  appended, decoded synchronously, selected).
- Custom: `+ Save current...` snapshots the current ModalParams +
  physics flag into a new entry, opens it in inline-rename mode.
- Right-click on any user-added entry → Rename / Remove.

Persistence: user-added entries (only the ones with `builtin =
false`) are written to `~/.keysynth/voices.json` (or
`%USERPROFILE%\.keysynth\voices.json`) as a `serde_json` blob on
every add / remove / rename. Built-ins are seeded fresh from code
each launch so they stay in sync with the binary. Restored on next
boot via `VoiceLibrary::load`.

GM 128 picker isn't gone — it's now a `CollapsingHeader` in the
central pane that only renders when `engine == SfPiano`, so the
left panel stays focused on instrument selection.

ModalParams sliders moved from the top panel into the central pane
(again, only visible when `engine == PianoModal`). Top panel kept
for master / reverb / mix mode + the asset-load status line.

Verified:
- `cargo build --release --bin keysynth` clean (4 pre-existing
  `_mm_*csr` deprecation warnings only).
- `cargo test --release --lib` — 208 pass / 4 fail. The 4 failures
  are the same pre-existing `voices::piano_modal::tests::{from_lut_*,
  release_lifecycle}` ones documented above; not regressed.
- New unit tests in `voice_lib::tests` (3 cases) cover built-in
  population, built-in protection from removal, and round-trip
  add → select → remove of a Custom slot.
- Salamander SFZ auto-discovery (Salamander/SalamanderGrandPianoV3+20200602/
  *.sfz) and `GeneralUser-GS.sf2` both surface as Piano-category
  entries on startup so first-click is instant.
